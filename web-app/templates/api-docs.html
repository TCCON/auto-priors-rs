Authenticating

All the API endpoints for this site require that you authenticate your request. This requires that you have a token, which can be obtained from the token management page. For the rest of the documentation, we will assume that your token is saved to the file ~/.priors-api-key.

Important
Your tokens grant you access to the API, just like a username and password. Protect them just like you would your passwords: do not share them or send them between computer unencrypted (e.g., through email). If you need to copy a token from one computer to another, use something like scp or rsync (over SSH).

Jobs endpoints

submit

URL: /api/v1/jobs/submit

Description: Submit a request to generate priors for a custom location.

URL Parameters: None

Request type: POST

Request body: A JSON structure with the following fields:

  • start_date: the first date to produce priors for, in YYYY-MM-DD format
  • end_date: the date after the last date to produce priors for, in YYYY-MM-DD format
  • sites: a list of structures with keys "site_id" (two character site ID), "lat" (latitude in range -90 to +90), and "lon" (longitude in range -180 to +180). This defined where to generate priors. Multiple locations can be specified.
  • mod_fmt (optional): what format to generate .mod files in. Allowed values are "text" and "none". "text" is default.
  • vmr_fmt (optional): what format to generate .vmr files in. Allowed values are "text" and "none". "text" is default.
  • map_fmt (optional): what format to generate .map files in. Allowed values are "text", "netcdf", "textandnc", and "none". "text" is default.
  • reanalysis (optional): what set of input data to use when generating the priors. More information TBD.
  • is_egi (optional): a boolean indicating if this request is from EGI. Setting to true alters the output file name to aid EGI in finding the output. Requests manually submitted should not use this field.

A minimal example of the request body:

{
    "start_date": "2025-01-01",
    "end_date": "2025-01-02",
    "sites": [
        {
            "site_id": "aa",
            "lat": 12.34,
            "lon": -56.78
        }
    ]
}

An example input with two sites and adjusting the requested file formats:

{
    "start_date": "2025-01-01",
    "end_date": "2025-01-02",
    "sites": [
        {
            "site_id": "aa",
            "lat": 12.34,
            "lon": -56.78
        },
        {
            "site_id": "bb",
            "lat": -43.21,
            "lon": 87.65
        }
    ],
    "mod_fmt": "none",
    "vmr_fmt": "none",
    "map_fmt": "netcdf"
}

Returns: A JSON structure with three fields:

  • successful: a boolean indicating if the request was added to the queue
  • job_ids: a list of the job IDs that were submitted to the queue for this request. If the request failed, this will be null.
  • error_reason: a string giving the reason(s) that the request was rejected or otherwise failed to be added to the queue. If the request succeeded, this will be null.

Example:

cURL:

{{ Self::curl_post_example(self, "/api/v1/jobs/submit", "application/json", "{\"start_date\": \"2023-01-01\", \"end_date\": \"2023-01-02\", \"sites\": [{\"site_id\": \"aa\", \"lat\": 12.34, \"lon\": -56.78}]}") }}

Python:

{{ Self::py_post_json_example(self, "/api/v1/jobs/submit", "{\n    \"start_date\": \"2023-01-01\",\n    \"end_date\": \"2023-01-02\",\n    \"sites\": [\n        {\n            \"site_id\": \"aa\",\n            \"lat\": 12.34,\n            \"lon\": -56.78\n        }\n    ]\n}") }}

    "job_ids": [56678],

Bad request output:

If your request is received but cannot be fulfilled for some reason, you will get bad a "Bad Request" (status code 400). The JSON returned will provide more information in this case.

{
    "successful": false,
    "job_ids": null,
    "error_reason": "Request was invalid because:\nYour request could not be fulfilled: met data was unavailable for 1 of the dates requested: 2025-01-01."
}

Downloading

job

URL: /api/v1/download/job/{JOB_ID}

Description: Return the gzipped tarball of the given job's output, if available.

URL Parameters:

  • JOB_ID: the numeric ID of the job. This can be found with the active-jobs or all-jobs endpoints, or from the return value from a successful submit-job request.

Request type: GET

Request body: Nothing

Returns on success: the gzipped tarball as a file stream

Status code meanings:

  • OK (200) - success, file will be transmitted
  • BAD_REQUEST (400) - an error occurred while querying the job or the job does not have a tarball as its output. Usually, this means that you provided a job ID that does not exist, but could indicate a server error. Contact the priors system admins if you are certain that the job ID is correct and should exist.
  • FORBIDDEN (403) - you do not have permission to access this job.
  • GONE (410) - the output from this job has already been cleaned up, and will not be available.
  • NO_CONTENT (204) - the output file does not exist. Check the job status to see if the job is just not complete or has errored.

Example:

cURL:

Python:

{{ Self::py_download_example(self, "/api/v1/download/job/1") }}

job

URL: /api/v1/download/stdsite/{SITE_ID}/{DATE}

Description: Return the gzipped tarball the priors for the given site on the given date, if available.

URL Parameters:

  • SITE_ID: the two-character site or campaign ID, e.g. "pa" for Park Falls.
  • DATE: the UTC date for which you want the priors, in YYYY-MM-DD format, e.g. "2020-01-02" for 2 Jan 2020.

Request type: GET

Request body: Nothing

Returns on success: the gzipped tarball as a file stream

Status code meanings:

  • OK (200) - success, file will be transmitted
  • BAD_REQUEST (400) - either the site ID is incorrect, the date is in the incorrect format, or that date is has not been entered into the table of dates for that site yet. If the date is recent (within about one day), the request may succeed if send again later. Otherwise, this could mean the date is outside the range of dates for which this site has priors generated.
  • NO_CONTENT (204) - the output file does not exist yet, but will in the future.
